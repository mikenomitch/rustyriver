# T8 Porting Brief — `store.rs`: Snapshot/TXID Bookkeeping + Retention

> Ported from `litestream@v0.5.11 store.go` (895 LOC) and `compaction_level.go` (114 LOC).
> Upstream test surface: `store_test.go` (484 LOC), `store_compaction_remote_test.go` (277 LOC),
> `heartbeat_test.go` (256 LOC).

---

## 1. Purpose

`store.rs` is the top-level orchestration layer that owns the collection of
`Db` instances and drives background compaction, snapshot, and retention
tasks.  It enforces the multi-level compaction schedule (`CompactionLevel`
intervals), decides which LTX files survive retention cuts at each level, and
provides a single synchronous entry-point (`compact_db` / `enforce_snapshot_retention`) for external callers (e.g., the CLI and tests) to trigger those operations on demand.

Because the one-shot scope is **L0-only** (PLAN.md §2, D-7), the full
compaction-level machinery is deferred — but the snapshot monitor loop,
`SnapshotLevel = 9`, retention GC, and `SyncDB` helper are all **in scope**
as they are exercised by T10 (`replica.rs`) and required for G2 gate
conformance.

## 5. Footguns & Correctness-Critical Spots

### 5.1 Retention boundary comparisons — STRICT `<`, NOT `<=`

In `EnforceRetentionByTXID` (`compactor.go:305`):
```go
if info.MaxTXID < txID {  // strict less-than
    deleted = append(deleted, info)
}
```
Files **at exactly** `txID` are kept. Using `<=` silently deletes the anchor
file and can create a gap that breaks restore. The Rust port must use the same
`<` semantics.

In `EnforceSnapshotRetention` (`db.go:2081`):
```go
if info.CreatedAt.Before(timestamp) {  // strict Before (exclusive)
```
Files created **exactly at** the cutoff timestamp are retained.  Use
`SystemTime::lt` not `SystemTime::le`.

### 5.2 "Always keep the newest file" invariant

Both `EnforceSnapshotRetention` and `EnforceRetentionByTXID` share this
guard (compactor.go:263, db.go:2094):
```go
if len(deleted) > 0 && deleted[len(deleted)-1] == lastInfo {
    deleted = deleted[:len(deleted)-1]
}
```
This prevents the case where all files fall outside retention but no newer
file has been written yet — would leave the replica empty and unrestorable.
**Easy to forget when the list is long and the newest file happens to be
expired.**  Test with a single file whose timestamp is before the cutoff;
it must survive.

### 5.3 `ErrCompactionTooEarly` guard — time.Truncate semantics

`PrevCompactionAt` uses `time.Truncate(interval).UTC()` (`compaction_level.go:33`).
Go's `Truncate` floors to the nearest multiple of the interval relative to the
zero time (Unix epoch). Use `chrono::DateTime::duration_trunc` or equivalent
exact-same math — **do not use `Duration::div` on the elapsed time**; the
answers differ by sub-second offsets. The `TestStore_CompactDB/L1` test has a
documented edge case comment (store_test.go:52-60) that `ErrCompactionTooEarly`
**or** `ErrNoCompaction` is acceptable immediately after the first compaction,
depending on whether a second boundary was crossed mid-test. Port this OR
semantic exactly.

### 5.4 `PageSize == 0` guard before compaction

`CompactDB` (store.go:748) must return `DbNotReadyError` when `db.page_size() == 0`.
The regression test `TestStore_CompactDB/DBNotReady` is the canonical check.
This guard is a sentinel, not an error variant — it must be downcasting-comparable
(`errors::Is` in Go; implement `PartialEq` or a typed is-check in Rust).

### 5.5 Concurrent `register_db` double-open race

`RegisterDB` (store.go:267-320) uses a double-checked pattern: first check
under lock, release, open (potentially slow), re-acquire lock and check again.
The second instance is closed with `db.Close(context.Background())` even
though we hold no data about the winner. Rust: use `Mutex`-guarded `Vec` with
the same pre-check / post-check / close pattern; do not simplify to a single
lock-then-open (blocks all DB ops during slow init).

### 5.6 `monitorCompactionLevel` retry loop — `DBNotReady` vs timeout

`monitorCompactionLevel` (store.go:538-604) maintains a `retryDeadline` to
distinguish "retry soon because a DB is initializing" from "warn and give up."
The countdown (30 s default) is reset each time at least one DB is not ready.
If the deadline passes, the warning is logged and `retryDeadline` is cleared —
but the loop does **not** stop. Rust: store `Option<Instant>` for the deadline;
reset it to `None` after the timeout is logged.

### 5.7 Eventual consistency / partial-read vulnerability

`TestStore_CompactDB_RemotePartialRead` documents a real S3 corner case: an
object becomes visible in the listing before all bytes are available.  The
compactor reads the partial object and writes a truncated snapshot.  The fix is
defensive: after `WriteLTXFile`, a subsequent `OpenLTXFile` at full size must
succeed before committing the compacted level info to the cache.  The Rust port
should include the `delayedReplicaClient` mock test to prevent regression.

### 5.8 L0 retention stop-early semantic

`EnforceL0RetentionByTime` (db.go:2180-2193) stops iterating once it finds a
file newer than the threshold — it does **not** scan past it — because L0 files
are ordered and skipping a newer file would create a gap:
```go
if createdAt.After(threshold) {
    processedAll = false
    tooRecentCount++
    break   // ← stop here; do not mark subsequent older files for deletion
}
```
This "stop on first retained file" pattern is a common retention footgun —
naively deleting all files that satisfy the predicate without respecting order
creates a non-contiguous L0 set that breaks VFS read replicas.

### 5.9 `srcInfo.MaxTXID <= dstInfo.MinTXID` vs `<`

`CompactDB` (store.go:780) uses `<=` to detect "no progress":
```go
if srcInfo.MaxTXID <= dstInfo.MinTXID {
    return nil, ErrNoCompaction
}
```
This is distinct from the `< txID` in retention.  Keep both inequalities exact.

### 5.10 `Close` error priority — `ErrShutdownInterrupted` is lower priority

`Close` (store.go:236-258) collects errors from all DBs and returns the first
non-`ErrShutdownInterrupted` error, or `ErrShutdownInterrupted` only if no
other error occurred.  This priority logic must be faithfully reproduced —
do not collapse all errors into one with `?`.

## 7. Size & Risk

**Source to port:**
- `store.go`: 895 LOC
- `compaction_level.go`: 114 LOC
- Directly related `db.go` retention methods referenced but owned by T9: ~400 LOC
- `compactor.go` (compaction / retention logic, owned by T9): ~334 LOC

**T8's direct contribution:** ~400–500 Rust LOC (types, orchestration, monitors, retention algorithm; the heavy lifting is in T9's `Db` methods).

**Risk: MED**

Justification: the retention boundary comparisons (strict `<`, keep-newest
guard, L0 stop-early) are individually simple but collectively easy to get
subtly wrong, and a mistake silently discards data rather than causing an
immediate test failure. The eventual-consistency partial-read regression
(`TestStore_CompactDB_RemotePartialRead`) requires a non-trivial mock
`ReplicaClient` in Rust. The compaction-level interval math has a documented
edge case in the upstream tests. None of these are fundamentally hard, but
each needs deliberate attention — the overall complexity is medium.
