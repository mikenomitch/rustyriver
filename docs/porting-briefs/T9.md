## 1. Purpose

`db.rs` manages one live SQLite database file: it holds a **long-running read
transaction** so no other process can checkpoint/restart the WAL out from under
us (the "checkpoint takeover"), then on each `Sync` it diffs the real WAL against
the last LTX position and writes the new pages as the next **L0 LTX file**
(`db.go:1517-1723`). It owns the progressive 3-tier checkpoint policy (PASSIVE /
TRUNCATE, `db.go:1092-1156`), the snapshot-on-continuity-break decision
(`verify`, `db.go:1296-1436`), and a clean shutdown that flushes a final sync and
releases the read-tx without corruption (`Close`, `db.go:602-650`). It is the
local half of replication; the remote half (upload, restore) lives in T10
`replica.rs`, which this module calls through the `Replica` handle.

## 2. Rust API shape

Go's `DB` is one big struct with a `sync.RWMutex` (`mu`), a second checkpoint
`sync.RWMutex` (`chkMu`), plus several finer-grained mutexes. The natural Rust
shape is a `Db` struct whose mutable shared state lives behind `tokio::sync`
primitives, cloneable via `Arc<DbInner>`. **All public methods are `async`** (D-9
Tokio); SQLite calls must hop to a blocking context (see §5, footgun F-1).

**Trait vs struct.** `Db` is a concrete struct (Go has no `DB` interface). It
holds the `ReplicaClient` (T5) behind `Arc<dyn ReplicaClient>` (an `async-trait`
object) via the `Replica` handle. Tests inject mocks at that trait boundary
(§4). The four `CheckpointMode`s replace Go's stringly-typed `mode` param —
`execCheckpoint` interpolates the string straight into `PRAGMA wal_checkpoint(<mode>)`
(`db.go:1905`), so the enum must `Display` to exactly `PASSIVE`/`FULL`/`RESTART`/`TRUNCATE`.

**Async/blocking boundary.** Every `rusqlite` call is synchronous and the read-tx
is `!Send`-ish (a `rusqlite::Transaction` borrows its `Connection`). Two viable
shapes (escalate the choice in `OPEN_QUESTIONS.md` before coding — it shapes the
whole module):
- **(A) dedicated DB thread**: own the `Connection` + read-tx on one OS thread,
  drive it via a command channel; `async` methods `await` oneshot replies.
- **(B) `spawn_blocking` + `Mutex<Connection>`**: simpler, but the read-tx
  cannot be held across `await` while other code touches the same connection —
  you must keep the read-tx and all PRAGMA traffic serialized through one guard.
  `DECISION:` lean (A) for the read-tx lifecycle correctness this module needs.

## 4. Test surface

**Harness facts** (apply to every test):
- All three target files are **pure / needs-tempdir-fs**. **None** use MinIO or
  the real `litestream` binary (verified: no `exec.Command`/`minio`/`integration`
  refs in the three files). They use `rusqlite` + a tempdir + a *local* mock
  replica client. So in Rust: `tempfile::TempDir` + a fake `ReplicaClient`.
- `db_test.go` / `db_shutdown_test.go` are Go `package litestream_test` (black-box):
  port to `tests/db_*.rs` or a `#[cfg(test)] mod` using only the **public** `Db` API.
- `db_internal_test.go` is `package litestream` (**white-box**): it touches
  private fields (`db.rtx`, `db.pageSize`, `db.verify()`, `db.invalidatePosCache()`,
  `db.checkpoint()`, `db.sync` internals) and `prometheus/testutil`. **These must
  live in an in-module `#[cfg(test)] mod tests` inside `db.rs`** so they can reach
  private state. This split drives the sub-step plan (§6).
- Helper `testingutil.MustOpenDBAt` sets `MonitorInterval=0` (no background task)
  and `ShutdownSyncTimeout=0` for determinism (testingutil.go:185-186); the file
  client is `file.NewReplicaClient(tempdir)`. Port equivalent `must_open_db(s)` helpers.
- The mock is a struct of optional closures (`mock/replica_client.go:15-53`) →
  Rust: a `MockReplicaClient` holding `Box<dyn Fn..>` fields (or builder), behind
  the `ReplicaClient` trait. `db_internal_test.go` uses its own `testReplicaClient`
  (a minimal local-dir file client, db_internal_test.go:26-110) — reproduce as a
  small test helper.

## 5. Footguns & correctness-critical spots

**F-1 — read-tx lifecycle vs `rusqlite` (R-4, the central risk).** Go holds a
`*sql.Tx` open (`acquireReadLock` runs `BEGIN` + `SELECT COUNT(1) FROM
_litestream_seq`, db.go:956-976) to block external checkpoints. Before *every*
checkpoint it **releases** the read-tx, runs `PRAGMA wal_checkpoint(<mode>)`, and
**re-acquires** it (`execCheckpoint`, db.go:1892-1916, with a `defer` re-acquire
on early return). In `rusqlite` a `Transaction` borrows the `Connection`, so you
cannot hold it across `await` while another task uses that connection. Use a
dedicated connection/thread for the read-tx (shape A in §2). The exact
sequence — *release → checkpoint → re-acquire, re-acquire even on error* — is
load-bearing for `TestDB_releaseReadLock_DoubleRollback` and the concurrent-write
race. **Do not guess SQLite semantics — escalate (AGENTS.md §3.3 rule 9).**

**F-2 — `releaseReadLock` must swallow "already rolled back".** Go's `rollback`
helper ignores the `transaction has already been committed or rolled back`
string (litestream.go:130-135) because Close and a concurrent checkpoint can both
try to release (issue #934, db.go:986-991). In Rust, `rusqlite`'s
`Transaction::rollback()` consumed-self semantics or an explicit
`finish()`/drop must be made idempotent; a double release must return `Ok(())`,
not error (`TestDB_releaseReadLock_DoubleRollback`).

**F-3 — two distinct locks, do not collapse them.** `mu` (db.go:65) serializes
`Sync`/`Checkpoint`/`CRC64` public entry; `chkMu` (db.go:73) is a *separate*
checkpoint-vs-snapshot gate. `sync` takes `chkMu.RLock` (db.go:1546, **unless**
already checkpointing), `checkpoint` takes `chkMu.TryLock` and **returns early
(no-op) if it can't get it** (db.go:1812-1814) — i.e. a snapshot in progress
silently skips the checkpoint. `SnapshotReader` also `RLock`s chkMu (db.go:1936).
Collapsing these into one mutex will deadlock (checkpoint calls `verifyAndSync`
which calls `sync`, db.go:1824). Mirror as a `tokio::sync::RwLock<()>` used as a
gate, with `try_write()` for the checkpoint path.

**F-4 — TXID is strictly `pos.TXID + 1` per L0 file (db.go:1525).** Every sync
that produces output writes file `<txid>-<txid>.ltx` at L0 with `MinTXID ==
MaxTXID == txid` (db.go:1527,1642-1643). Snapshots span `MinTXID=1 ..
MaxTXID=pos.TXID` (db.go:1996-1997). The **position cache** (`pos_cache`,
db.go:106-109) is updated from the encoder's `post_apply_pos()` after a
successful atomic rename (db.go:1699-1702) and invalidated on any L0 delete /
rename failure / reset (db.go:357,430,1683). Ordering bug here cascades into
restore.

**F-5 — the three anti-feedback-loop flags (issues #896/#927/#997).** These are
subtle and individually tested:
- `synced_since_checkpoint` (db.go:80): time-based checkpoints only fire if data
  was synced since last checkpoint, else idle DBs spin LTX files (#896). Set true
  in `sync` after a real sync (db.go:1028-1030); cleared in `checkpoint`
  (db.go:1840,1871).
- `synced_to_wal_end` (db.go:88): if the last sync reached exact WAL EOF, the
  *next* truncation is expected (checkpoint), so `verify` resets to header and
  continues incrementally instead of snapshotting (#927, db.go:1335-1355). Set at
  db.go:1714-1718.
- `last_synced_wal_offset` (db.go:96): checkpoint thresholds use the **logical**
  WAL offset (WALOffset+WALSize from the last LTX), **not file size**, because
  stale post-checkpoint frames with old salts inflate file size and cause endless
  PASSIVE checkpoints (#997, db.go:1058-1089,1704-1709). Get this wrong and
  `TestDB_IdleCheckpointSnapshotLoop` / `TestDB_Issue994` fail.

**F-6 — error-string classification.** `isSQLiteBusyError` / `isDiskFullError`
match on substrings `"database is locked"`, `"SQLITE_BUSY"`, `"no space left on
device"`, etc. (db.go:1158-1180). PASSIVE checkpoints that hit BUSY are **not
errors** — log-and-continue (db.go:1118-1124). In Rust prefer matching
`rusqlite::ErrorCode::DatabaseBusy` / `ffi::SQLITE_BUSY` for the busy case and
`io::ErrorKind`/raw errno (`ENOSPC`/`EDQUOT`) for disk-full, but keep the
substring fallback for parity (`TestIsSQLiteBusyError`/`TestIsDiskFullError`).
NB: this is the one log-and-continue case the upstream CLAUDE.md explicitly
sanctions (best-effort, doesn't affect correctness).

**F-7 — `verify()` is the snapshot-on-continuity-break brain (db.go:1296-1436).**
Decision lattice it must reproduce exactly: first sync (pos.TXID==0) → offset=32;
LTX offset > WAL size → either expected-truncation reset (if `synced_to_wal_end`)
or `reason="wal truncated by another process"`; offset==32 edge (avoid the
`32-4120` underflow, #900) handled *before* computing `prevWALOffset`; salt
compare; `lastPageMatch` (re-read the last synced page and find it in the last
LTX, db.go:1438-1475) to decide overwrite-vs-not; salt changed + last-page-match
→ `detectFullCheckpoint` (count unknown salts via `frame_salts_until`,
db.go:1477-1507). Each branch has a test (§4). **This is a correctness-critical
path — port branch-for-branch, do not refactor for "elegance" on pass one**
(PLAN.md §11.1).

**F-8 — atomic LTX write + ownership.** Write to `<name>.tmp`, `chown` to the DB
file's uid/gid, `fsync`, close, then `rename` to final; on rename failure clear
the L0 cache entry and invalidate pos (db.go:1609-1685). The temp dir is created
with the parent dir's perms (`internal.MkdirAll(dir, dirInfo)`, db.go:1610). Use
`tempfile`/explicit `.tmp` + `fs::rename`; preserve fsync-before-rename for crash
consistency (T14 leans on this). `removeTmpFiles` runs on Open to clear crash
leftovers (db.go:576, litestream.go:169-182).

**F-9 — checkpoint two-phase WAL-restart handling (db.go:1810-1873).** After
`execCheckpoint` it writes one `_litestream_seq` row to force a fresh WAL page,
re-reads the WAL header; if **unchanged** it's done; if **changed** (WAL
restarted) it opens a write-tx (via `INSERT INTO _litestream_lock` to grab the
write lock without `BEGIN IMMEDIATE`, db.go:1845-1857), does a post-checkpoint
`verifyAndSync(checkpointing=true)`, then rolls back. The TRUNCATE path is what
`TestDB_CheckpointCreatesSnapshotL0` and the page-gap race exercise. The
`_litestream_seq` / `_litestream_lock` tables are created in `init` (db.go:857-865).

**F-10 — init sequencing & PERSIST_WAL.** `init` (db.go:795-911): stat db + parent
(cache `fileInfo`/`dirInfo` for perms), open with DSN
`busy_timeout(<ms>)&wal_autocheckpoint(0)` (db.go:818 — **autocheckpoint MUST be
0**; we own checkpointing), set `PERSIST_WAL` file-control so SQLite doesn't
delete the WAL on last close (db.go:771-793,825-828), open a separate long-lived
**read-only file descriptor** on the DB (needed for non-OFD locks + `ReadAt`,
db.go:830-833), assert `journal_mode=wal`, create the two internal tables,
acquire read lock, read page size, `ensureWALExists` (force ≥1 frame,
db.go:1199-1209), `checkDatabaseBehindReplica` (#781, db.go:1211-1294), then
`Replica.Start`. `rusqlite` exposes `pragma_update`/`busy_timeout`; PERSIST_WAL
needs `unsafe` FFI (`sqlite3_file_control` with `SQLITE_FCNTL_PERSIST_WAL`) —
**this is the most likely `unsafe` block in the crate; justify it per AGENTS.md
§3.2 rule 6, or find a safe rusqlite path and log the decision.**

**F-11 — `checkDatabaseBehindReplica` restore-detection (db.go:1217-1294).** If
local L0 max TXID < replica max TXID (a fresh-host restore scenario), it clears
local L0, fetches the replica's latest L0 to a temp file, atomically renames, and
invalidates pos — so the next sync snapshots at current DB state. Needs the
`Replica.MaxLTXFileInfo` + `ReplicaClient.OpenLTXFile` surface (T10/T5).

**F-12 — shutdown sync retry & `Done` interruption (db.go:602-768).** `Close`
order is exact: `cancel()` the monitor → `wg.Wait()` → final `Sync` → 
`syncReplicaWithRetry` → `Replica.Stop(hard=true)` → `releaseReadLock` → close
conn → close fd → mark `opened=false`, `rtx=None`. `syncReplicaWithRetry`
(db.go:656-768): `timeout==0` → single attempt (no retry); else deadline ctx +
optional `Done`-derived cancel that interrupts in-flight `Replica.Sync`;
exponential nothing here (linear `interval` waits); returns
`ErrShutdownInterrupted` on `Done`, `"timeout"`-wrapped error on deadline. Map Go
ctx+channels to `tokio::select!` over `CancellationToken`s + `tokio::time`. All
eight shutdown subtests pin these timings/attempt-counts.

**F-13 — monitor loop backoff (db.go:2250-2320).** Background task: ticker at
`MonitorInterval`; on sync error increments `consecutiveErrs`, doubles `backoff`
(seed = MonitorInterval) capped at `DefaultSyncBackoffMax=5m`; rate-limits error
logs to every `SyncErrorLogInterval=30s`; on ≥3 disk-full errors tries
`removeTmpFiles`; resets on success. Tests disable it (`MonitorInterval=0`), but
`TestDB_ConcurrentMapWrite` turns it on at 10ms — so the task must be cancellable
and not race the public `sync`.

**F-14 — retention boundary contiguity (db.go:2118-2242, 2063-2116).** L0
retention deletes a file only if it's **both** compacted into L1 (`MaxTXID <=
maxL1TXID`) **and** older than `L0Retention`; it **stops at the first too-recent
file** to keep coverage contiguous for VFS (db.go:2180-2186), and never deletes
the newest file when the whole level was processed (db.go:2202-2205).
Snapshot retention always keeps ≥1 snapshot (db.go:2093-2096). `RetentionEnabled
== false` → skip remote delete but **still delete local** (db.go:2099-2113). Off-
by-one here drops a needed file. (L1 compaction itself is DEFER scope, but L0
retention is in scope and tested.)

**F-15 — `SnapshotReader` streams concurrently (db.go:1922-2021).** Go spawns a
goroutine writing into an `io.Pipe`. In Rust: a `tokio::spawn` writing into a
`tokio::io::duplex`/`DuplexStream`, or build the LTX fully then return a cursor.
Propagate encode errors to the reader (Go uses `pw.CloseWithError`). Keep the
`chkMu.RLock` held for the duration so no checkpoint races the read.

## 6. Suggested sub-step breakdown

Each sub-step lands with the **full gate green** (`cargo fmt --check && cargo
clippy -- -D warnings && cargo build --all-targets && cargo test --all`,
AGENTS.md §3.2) and only the tests it owns enabled; later sub-steps un-skip more.
Ordered to put the read-tx/SQLite spine first and the hardest race last.

**S9.0 — skeleton + SQLite/read-tx spine (no replication).**
`Db`/`DbConfig`/`CheckpointMode`/`SyncStatus`/`SyncInfo` types; `new`, path
helpers, `open`/`close` minus replica calls; `init` (DSN, PERSIST_WAL via the
F-10 `unsafe`/decision, read-only fd, internal tables), `acquireReadLock` /
`releaseReadLock` (idempotent, F-2), `ensureWALExists`, `calc_wal_size`, the
error sentinels (§3). **Decide the async/blocking shape (§2) here.** *Gate tests:*
`Path/WALPath/MetaPath`, `TestCalcWALSize`, `TestDB_releaseReadLock_DoubleRollback`,
`TestIsSQLiteBusyError`/`TestIsDiskFullError`. Mock replica is a no-op.

**S9.1 — `pos()` + capture loop happy path (`sync`/`verify`/`sync` inner).**
`pos()` with cache + LTXError mapping; `verify` full branch lattice (F-7);
`writeLTXFromWAL` + `writeLTXFromDB`; atomic tmp→rename + cache update (F-8);
first-sync snapshot. Wire `WalReader`/`ltx::Encoder`/`Decoder` (T1/T2). *Gate
tests:* `TestDB_Sync{NoDB,Initial,MultiSync,NoWAL,OverwritePrevPosition}`,
`TestDB_Verify_WALOffsetAtHeader{,_SaltMismatch}`,
`TestDB_Pos_{Open,Verify}ErrorReturnsLTXError`, `TestDB_SyncAfterVacuum`,
`TestDB_WALPageCoverage*`, `TestDB_WriteLTXFromWAL_PageGrowthCoverage`.

**S9.2 — checkpoint policy + two-phase restart + anti-loop flags.**
`checkpoint`/`execCheckpoint` (release→PRAGMA→re-acquire, F-1), `chkMu` gate
(F-3), WAL-restart write-tx path (F-9), `checkpointIfNeeded` (3-tier), and the
`synced_since_checkpoint`/`synced_to_wal_end`/`last_synced_wal_offset` flags
(F-5). *Gate tests:* `TestDB_Sync{MinCheckpointPageN,TruncatePageN}` (keep
`CheckpointInterval` `#[ignore]`-equivalent **only because upstream skips it** —
note in coverage report), `TestDB_CheckpointDoesNotTriggerSnapshot{Truncate,Passive}`,
`TestDB_MultipleCheckpointsWithWrites`, `TestDB_NoLTXFilesOnIdleSync`,
`TestDB_DelayedCheckpointAfterWrite`, `TestDB_IdleCheckpointSnapshotLoop`,
`TestDB_Issue994_RunawayDiskUsage`, `TestDB_CheckpointCreatesSnapshotL0`.

**S9.3 — snapshot + CRC64 + `ResetLocalState`.** `SnapshotReader` (streaming,
F-15), `Snapshot` (writes snapshot-level LTX via `ReplicaClient.WriteLTXFile`),
`CRC64` (RESTART + full-file CRC64-ISO), `reset_local_state`, `invalidatePosCache`.
*Gate tests:* `TestDB_Snapshot` (golden-critical CRC equality), `TestDB_CRC64`,
`TestDB_ResetLocalState`.

**S9.4 — replica integration: `EnsureExists`, `SyncStatus`, `SyncAndWait`,
retention.** Bind to the `Replica`/`ReplicaClient` surface (§3); if T10/T8 lag,
stub behind the trait with a test fake. `EnsureExists` (restore-if-missing +
swallow no-snapshot, F-11 `checkDatabaseBehindReplica`), `enforce_snapshot_retention`,
`enforce_l0_retention_by_time` (F-14 boundaries). *Gate tests:* `TestDB_EnsureExists*`,
`TestDB_SyncStatus*`, `TestDB_SyncAndWait*`, `TestDB_EnforceRetention`,
`TestDB_EnforceSnapshotRetention_RetentionDisabled`,
`TestDB_EnforceL0RetentionByTime{,_RetentionDisabled}`,
`TestDB_EnforceRetentionByTXID_LocalCleanup`.

**S9.5 — shutdown semantics: `syncReplicaWithRetry` + `Done`.** The retry loop,
`Done`/second-Ctrl+C interruption, zero-timeout single attempt, context
cancellation (F-12), and the monitor backoff loop (F-13). *Gate tests:* all 8
`TestDB_Close_SyncRetry` subtests.

**S9.6 — concurrency hardening + race tests.** Turn on the monitor task; verify
no `Sync`/`Snapshot`/checkpoint data races; run the page-gap race. Add a `loom`
model of the read-tx acquire/release/re-acquire + chkMu gate where feasible
(PLAN.md §6.4 "Concurrency within T9"); `loom` can't drive real SQLite, so the
real-SQLite races stay as `--test-threads`/TSan integration tests. *Gate tests:*
`TestDB_ConcurrentMapWrite`, `TestDB_CheckpointPageGapWithConcurrentWrites`,
`TestDB_Sync_CompactionValidAfterGrowthAndCheckpoint`.

## 7. Size & risk

- **Upstream `db.go`:** 2489 LOC. Tests: `db_test.go` 1708 + `db_internal_test.go`
  2127 + `db_shutdown_test.go` 398 = 4233 LOC of tests.
- **Estimated Rust module `db.rs`:** ~1300-1600 LOC implementation (async wrappers
  + the SQLite-thread shape add bulk over Go), plus ~1500-2000 LOC of ported tests
  (some compaction/metrics tests dropped/deferred). Call it **~1500 LOC** for the
  module proper.
- **Risk: HIGH.** It is the explicit R-4 hot-spot: a long-running `rusqlite`
  read-tx interacting with manual `PRAGMA wal_checkpoint` across the async
  boundary, with at least one likely `unsafe` FFI call (PERSIST_WAL), a
  branch-dense `verify()` continuity lattice, three subtle anti-feedback flags,
  and a real-SQLite concurrent-checkpoint race test that has no `loom` shortcut.
  Mitigations: incremental sub-steps with green gates (§6), `loom` on the lock
  protocol, and the differential/property layers (G2/G3) catching any LTX-output
  drift the unit tests miss.
