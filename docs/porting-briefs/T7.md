# T7 Porting Brief — `client/object_store.rs` (S3/R2 via `object_store`)

**Target file:** `src/client/object_store.rs`
**Upstream spec:** `reference/litestream-go/s3/replica_client.go` (1715 LOC) +
`reference/litestream-go/s3/replica_client_test.go` (2086 LOC)
**Depends on:** T5 (`ReplicaClient` trait in `client/mod.rs`)
**Preconditions:** Gate G1 (format) green; `ltx::FileInfo`, `ltx::TXID`, and the
`ReplicaClient` trait must all exist before this module can compile.

---

## 1. Purpose

`client/object_store.rs` is the S3/R2/MinIO backend for `rustyriver`.  It
implements the `ReplicaClient` trait using the `object_store` crate (Cargo
feature `s3`) instead of the Go AWS SDK, which is intentionally not ported.  The
module's job is to map the five `ReplicaClient` operations — list, read (with
byte-range), write, batch-delete, and delete-all — onto `object_store`'s
`ObjectStore` API, while preserving all the behavioral invariants (key/path
scheme, multipart thresholds, pagination, timestamp metadata, and error
semantics) that allow `rustyriver` to pass the T5 conformance suite against a
real MinIO instance.

---

## 2. Rust API Shape

The `object_store` crate provides a single `ObjectStore` trait for all
backends.  The porter does **not** talk to the AWS SDK; the credential and
endpoint wiring is handled by `object_store::aws::AmazonS3Builder` (behind the
`s3` feature).

```rust
// ── Key public types ─────────────────────────────────────────────────────────

/// Configuration for the S3/R2/MinIO backend.
///
/// Maps to the public fields of Go's `ReplicaClient` struct
/// (s3/replica_client.go:78-116).
#[derive(Debug, Clone)]
pub struct ObjectStoreConfig {
    pub bucket:              String,
    pub path:                String,         // prefix within the bucket
    pub region:              Option<String>,
    pub endpoint:            Option<String>, // custom endpoint (MinIO, R2, …)
    pub access_key_id:       Option<String>,
    pub secret_access_key:   Option<String>,
    pub force_path_style:    bool,
    pub skip_verify:         bool,
    // Upload tuning — zero means "use object_store defaults"
    pub part_size:           Option<u64>,    // default 5 MiB threshold
    pub concurrency:         Option<usize>,  // default 5; R2 default 2
}

impl ObjectStoreConfig {
    /// Construct from a parsed `s3://` URL, mirroring
    /// `NewReplicaClientFromURL` (s3/replica_client.go:132-313).
    pub fn from_url(url: &url::Url) -> crate::Result<Self>;
}

/// Concrete S3/R2/MinIO backend, wrapping an `Arc<dyn ObjectStore>`.
///
/// Mirrors Go `ReplicaClient` (s3/replica_client.go:78-116).
/// The inner store is created lazily on the first call that needs it
/// (mirrors `Init`, s3/replica_client.go:322-477).
pub struct ObjectStoreClient {
    store:  tokio::sync::OnceCell<Arc<dyn object_store::ObjectStore>>,
    config: ObjectStoreConfig,
}

impl ObjectStoreClient {
    pub fn new(config: ObjectStoreConfig) -> Self;

    /// Force early initialisation; idempotent (mirrors `Init`,
    /// s3/replica_client.go:322-477).
    pub async fn init(&self) -> crate::Result<()>;
}

// ── trait impl (the core contract) ───────────────────────────────────────────

// Go interface: reference/litestream-go/replica_client.go:18-51
#[async_trait::async_trait]
impl crate::client::ReplicaClient for ObjectStoreClient {
    /// Returns `"s3"`.
    fn client_type(&self) -> &str;

    /// ListObjectsV2 with level prefix; seek skips entries with
    /// `min_txid < seek` (s3/replica_client.go:601-611, 1360-1565).
    async fn ltx_files(
        &self,
        level: u32,
        seek:  crate::TXID,
        use_metadata: bool,
    ) -> crate::Result<impl Stream<Item = crate::Result<crate::ltx::FileInfo>>>;

    /// GET with a `Range: bytes=<offset>-[<end>]` header
    /// (s3/replica_client.go:613-653).
    async fn open_ltx_file(
        &self,
        level:    u32,
        min_txid: crate::TXID,
        max_txid: crate::TXID,
        offset:   u64,
        size:     u64,  // 0 = "to end of file"
    ) -> crate::Result<Box<dyn tokio::io::AsyncRead + Send + Unpin>>;

    /// PUT with optional multipart (threshold: 5 MiB)
    /// (s3/replica_client.go:655-727).
    /// Peeks the LTX header to extract the `Timestamp` field and stores it
    /// as `object_store::Attribute` (equivalent of the
    /// `litestream-timestamp` S3 metadata key, s3/replica_client.go:53).
    async fn write_ltx_file(
        &self,
        level:    u32,
        min_txid: crate::TXID,
        max_txid: crate::TXID,
        data:     impl tokio::io::AsyncRead + Send + Unpin,
    ) -> crate::Result<crate::ltx::FileInfo>;

    /// Batch DELETE up to 1000 keys per call
    /// (s3/replica_client.go:1027-1101).
    async fn delete_ltx_files(
        &self,
        files: &[crate::ltx::FileInfo],
    ) -> crate::Result<()>;

    /// LIST everything under the path prefix and batch-DELETE
    /// (s3/replica_client.go:1103-1148).
    async fn delete_all(&self) -> crate::Result<()>;
}
```

**Go → Rust type mapping**

| Go | Rust |
|----|------|
| `ltx.TXID` (`uint64`) | `crate::TXID` (`u64` newtype) |
| `*ltx.FileInfo` | `crate::ltx::FileInfo` |
| `ltx.FileIterator` | `Stream<Item=Result<FileInfo>>` (futures `Stream`) |
| `io.ReadCloser` | `Box<dyn AsyncRead + Send + Unpin>` |
| `io.Reader` | `impl AsyncRead + Send + Unpin` |
| `*s3.Client` | `Arc<dyn object_store::ObjectStore>` |
| `manager.Uploader` | `object_store` multipart via `put_multipart` |
| `context.Context` | Tokio `CancellationToken` threaded through, or just `&mut impl Future` cancellation |
| `slog.Logger` | `tracing::Span` / `tracing::instrument` |

The `use_metadata` flag on `ltx_files` has no direct `object_store` analogue;
the porter should implement `use_metadata = true` by calling
`ObjectStore::head` on each key to read back the `litestream-timestamp`
attribute (expensive; only used on restore paths).  See §5 for the
parallelism footgun.

**Key-path construction** — the exact S3 key pattern the implementation
*must* produce (wire-compatibility requirement D-1):

```
{path}/{level:04x}/{min_txid}-{max_txid}.ltx
```

where `{level:04x}` is the zero-padded 4-digit hex level, and both TXIDs are
16-digit zero-padded lowercase hex.  Example at level 0, TXID 1:
`replica/0000/0000000000000001-0000000000000001.ltx`

Source: s3/replica_client.go:629, 677, 1040-1042.

## 5. Footguns & Correctness-Critical Spots

### 5.1 Multipart threshold (CONF-critical)

The Go uploader switches to multipart at **5 MiB** (s3/replica_client.go:99,
351-465 in tests).  The `object_store` crate uses its own internal threshold;
the porter must verify the `put_multipart` vs `put` boundary matches 5 MiB or
configure it explicitly.  Mismatch causes test
`TestReplicaClient_MultipartUploadThreshold` to fail.

**object_store note:** `AmazonS3Builder::with_multipart_threshold` (or the
equivalent `ClientOptions` field) controls this.  Always set it explicitly to
`5 * 1024 * 1024` bytes to match Go.

### 5.2 `aws-chunked` / chunked transfer encoding

Multiple Go tests (test:169, test:239, test:292-308, test:460) assert that
`Transfer-Encoding: aws-chunked` and `X-Amz-Decoded-Content-Length` are
**absent** for any custom endpoint.  The `object_store` crate does not use
aws-chunked by default, so this is generally fine, but the porter should
verify no middleware re-enables it for MinIO.

### 5.3 Timestamp metadata and the `use_metadata` path

Go stores the LTX header's `Timestamp` field (milliseconds since epoch) in S3
object metadata as `litestream-timestamp: <RFC3339Nano>` on every PUT
(s3/replica_client.go:679-683).  On list, when `use_metadata = true`, it
makes a parallel batch of `HeadObject` calls (up to 50 concurrent, controlled
by `MetadataConcurrency`, s3/replica_client.go:63-64, 1384-1455) to retrieve
the actual LTX-header timestamp rather than S3 `LastModified`.

In `object_store` terms: store the timestamp as a custom attribute on `put`.
On `list + use_metadata`, call `head` for each object in a `FuturesUnordered`
or `buffer_unordered` with concurrency 50.  **Risk:** if the Rust port omits
the `head` fan-out and always uses `last_modified`, timestamp-based restores
will return incorrect results when many LTX files share the same wall-clock
second.

### 5.4 Pagination cursor correctness

`object_store` list is a `Stream`, not a paginator; the porter does not need to
manage a continuation token.  However, the seek-skip logic (s3/replica_client.go:
1530-1533) that skips objects with `min_txid < seek` must be applied as a
stream filter **after** parsing each key, not via a list prefix, because S3
prefix listing is lexicographic on the full key.  A bug here silently delivers
too-old LTX files and causes the store to replay old transactions.

### 5.5 Delete semantics — S3 returns per-key errors inside a 200 OK

The Go `DeleteObjects` call (s3/replica_client.go:1054-1098) checks `out.Errors`
*in addition to* the HTTP-level error.  `object_store` wraps this for the `delete`
path, but the batch-delete (`delete_objects`) method of `object_store` returns
individual errors.  The porter must check each entry in the batch result and
surface errors rather than silently swallowing partial failures.

### 5.6 `isNotExists` → `ErrorKind::NotFound` mapping

Go maps `NoSuchKey` (s3/replica_client.go:1662-1668) to `os.ErrNotExist`.  In
Rust, `object_store` already maps 404 to `object_store::Error::NotFound`.  Map
that to `crate::Error::Io(io::Error::from(io::ErrorKind::NotFound))` (or the
`NotFound` variant if T4 adds one) so callers using `std::io::ErrorKind::NotFound`
continue to work.

### 5.7 Concurrent init (OnceCell)

`Init` in Go acquires `mu.Lock()` (s3/replica_client.go:322-325) to initialise
exactly once.  Use `tokio::sync::OnceCell` for the `Arc<dyn ObjectStore>` field
so `init` is race-free without a `Mutex`.

### 5.8 Cloudflare R2 concurrency default

R2 has strict per-bucket concurrent multipart part limits.  Go defaults
`Concurrency = 2` for R2 endpoints (s3/replica_client.go:65-66, 264-265).
The `object_store` multipart method should be configured with
`concurrency = 2` when the endpoint matches `*.r2.cloudflarestorage.com`.
Test `TestReplicaClient_R2ConcurrencyDefault` (test:1792) exercises this.

### 5.9 Path-style URL requirement for MinIO / Backblaze / Supabase / Filebase

`AmazonS3Builder::with_virtual_hosted_style_request(false)` (or equivalent)
must be activated for any endpoint that is not native AWS S3
(s3/replica_client.go:258-263).

### 5.10 TXID ordering and retention boundaries

The list iterator must yield files in **ascending TXID order** (monotone).
`object_store` list is defined to return keys in lexicographic order; since
TXID hex strings are zero-padded to 16 digits, lexicographic equals numeric
order.  **Do not sort the stream post-hoc** — that would buffer the entire
listing.  Assert or document the invariant.  A violated ordering means the
store's retention-window calculation in T8/T10 silently deletes the wrong files.

## 7. Size & Risk

**Estimated implementation LOC:** ~450–600 lines of Rust (the Go file is 1715
LOC, but roughly half is AWS SDK middleware plumbing that `object_store`
eliminates; the content MD5 XML-marshal path is also not needed because
`object_store`'s delete API handles that internally).

**Risk: MED**

Justification: the `object_store` crate maps cleanly to the five `ReplicaClient`
methods and its AWS backend is well-tested against real S3 and MinIO.  The
non-trivial risks are (a) the multipart threshold boundary requiring explicit
configuration, (b) the `use_metadata` head fan-out whose concurrency and
error-handling must match the Go behaviour, and (c) the provider-specific
defaults table (R2 concurrency, path-style flags, endpoint detection) which is
purely a porting faithfulness concern.  None of these requires cryptographic
correctness or format parsing; mistakes produce observable test failures rather
than silent data loss.
