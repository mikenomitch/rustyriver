# T5 Porting Brief — `client/mod.rs`: `ReplicaClient` trait + generic conformance suite

**Source:** `reference/litestream-go/replica_client.go` (lines 1–161)
**Test spec:** `reference/litestream-go/replica_client_test.go` (lines 1–962)
**Depends on (must land before T5 starts):** T2 (`ltx.rs` — LTX read/write, `TXID`, `FileInfo`), T4 (`error.rs` — `Error`, `LTXError`, `TXID`, `Pos`, `Checksum`)
**Feeds (must pass conformance before proceeding):** T6 (`client/file.rs`), T7 (`client/object_store.rs`)

---

## 1. Purpose

`client/mod.rs` defines the single `ReplicaClient` trait that every storage backend must implement, plus a set of free-standing utility functions (`FindLTXFiles`, `FetchLTXHeader`, `FetchPageIndex`, `FetchPage`) that compose on top of that trait. Its most important deliverable for the overall test strategy is the **generic conformance suite** — a `run_client_suite(client)` function (or Rust equivalent via a macro / generic helper) that T6 and T7 both invoke with their concrete client types, ensuring every backend satisfies the same invariants. The suite is the primary mechanism by which "file client" and "S3 client" are proven interchangeable.

## 2. Rust API Shape

The types below are the target public surface. All async methods require `+ Send + Sync + 'static` bounds on the trait. The file is `src/client/mod.rs`.

```rust
// ── Go → Rust type mapping ────────────────────────────────────────────────
//
//   ltx.TXID             → crate::TXID  (u64 newtype, already in lib.rs)
//   ltx.FileInfo         → crate::ltx::FileInfo  (T2 owns this)
//   ltx.FileIterator     → crate::ltx::FileIterator  (T2 owns this)
//   ltx.Header           → crate::ltx::Header  (T2 owns this)
//   ltx.PageHeader       → crate::ltx::PageHeader  (T2 owns this)
//   ltx.PageIndexElem    → crate::ltx::PageIndexElem  (T2 owns this)
//   *slog.Logger         → tracing spans (no logger parameter — use tracing macros)
//   io.ReadCloser        → tokio::io::AsyncRead + Send (boxed) — or bytes::Bytes stream
//   io.Reader            → tokio::io::AsyncRead + Send (boxed)
//
// Note: the Go interface passes `*slog.Logger` into SetLogger; Rust uses tracing
// spans and the backend captures a logger internally if needed. Drop SetLogger.
//
// Level constants
pub const SNAPSHOT_LEVEL: i32 = 9; // SnapshotLevel = 9 (compaction_level.go:9)
pub const DEFAULT_ESTIMATED_PAGE_INDEX_SIZE: i64 = 32 * 1024; // replica_client.go:88

// ── ErrStopIter sentinel ─────────────────────────────────────────────────
// Ported from replica_client.go:16
pub struct StopIter;  // used as a signal value in FindLTXFiles filter closures

// ── Core trait ───────────────────────────────────────────────────────────
// Ported from ReplicaClient interface, replica_client.go:19–51
#[async_trait::async_trait]
pub trait ReplicaClient: Send + Sync {
    /// Returns a short string identifying the backend type (e.g., "file", "s3").
    /// replica_client.go:21
    fn client_type(&self) -> &'static str;

    /// Establishes connections, validates config. Must be idempotent.
    /// replica_client.go:24–26
    async fn init(&self, ctx: &Context) -> crate::Result<()>;

    /// Returns an async iterator of LTX FileInfo records at `level`,
    /// starting from `seek` (or the next available TXID if `seek` not present).
    /// `use_metadata` controls whether accurate CreatedAt timestamps are
    /// fetched (true = PITR path; false = fast LastModified).
    /// replica_client.go:29–32
    async fn ltx_files(
        &self,
        level: i32,
        seek: crate::TXID,
        use_metadata: bool,
    ) -> crate::Result<Box<dyn crate::ltx::FileIterator + Send>>;

    /// Opens a byte-range read on an LTX file.
    /// `offset` = byte offset into the file; `size` = max bytes to read (0 = read to EOF).
    /// Returns `Err(crate::Error::Io(io::ErrorKind::NotFound))` for missing files
    /// (maps to os.ErrNotExist in Go — replica_client_test.go:195).
    /// replica_client.go:34–37
    async fn open_ltx_file(
        &self,
        level: i32,
        min_txid: crate::TXID,
        max_txid: crate::TXID,
        offset: i64,
        size: i64,
    ) -> crate::Result<Box<dyn tokio::io::AsyncRead + Send + Unpin>>;

    /// Writes an LTX file from the supplied reader and returns its FileInfo
    /// (with Size and CreatedAt populated).
    /// replica_client.go:39–41
    async fn write_ltx_file(
        &self,
        level: i32,
        min_txid: crate::TXID,
        max_txid: crate::TXID,
        r: Box<dyn tokio::io::AsyncRead + Send + Unpin>,
    ) -> crate::Result<crate::ltx::FileInfo>;

    /// Deletes the specified LTX files in one call.
    /// replica_client.go:43–44
    async fn delete_ltx_files(
        &self,
        files: &[crate::ltx::FileInfo],
    ) -> crate::Result<()>;

    /// Deletes all files managed by this client (used in tests for teardown).
    /// replica_client.go:46–47
    async fn delete_all(&self) -> crate::Result<()>;
}

// ── Free-standing utility functions ──────────────────────────────────────

/// Returns all FileInfo records at `level` for which `filter` returns true.
/// Passing `StopIter` error from `filter` ends iteration cleanly.
/// replica_client.go:53–84
pub async fn find_ltx_files<C, F>(
    client: &C,
    level: i32,
    use_metadata: bool,
    filter: F,
) -> crate::Result<Vec<crate::ltx::FileInfo>>
where
    C: ReplicaClient,
    F: FnMut(&crate::ltx::FileInfo) -> Result<bool, crate::Error> + Send;

/// Reads and returns the LTX header for `info` by fetching exactly HeaderSize bytes.
/// replica_client.go:101–112
pub async fn fetch_ltx_header<C: ReplicaClient>(
    client: &C,
    info: &crate::ltx::FileInfo,
) -> crate::Result<crate::ltx::Header>;

/// Fetches and decodes the page index from the tail of an LTX file.
/// Uses a two-pass strategy: first fetches the last 32 KB; if the full
/// index was not included in that window, fetches again from the exact offset.
/// replica_client.go:90–98 (FetchPageIndex), 114–145 (fetchPageIndexData)
pub async fn fetch_page_index<C: ReplicaClient>(
    client: &C,
    info: &crate::ltx::FileInfo,
) -> crate::Result<std::collections::HashMap<u32, crate::ltx::PageIndexElem>>;

/// Fetches a single page frame (header + raw page bytes) at `(level, min_txid, max_txid)`
/// starting at `offset` with `size` bytes.
/// replica_client.go:148–160
pub async fn fetch_page<C: ReplicaClient>(
    client: &C,
    level: i32,
    min_txid: crate::TXID,
    max_txid: crate::TXID,
    offset: i64,
    size: i64,
) -> crate::Result<(crate::ltx::PageHeader, Vec<u8>)>;

// Called by T6 and T7 tests with their concrete client.
// The Go pattern is RunWithReplicaClient (replica_client_test.go:230–245)
// wrapping each test. The Rust equivalent should be a free function (or macro)
// that each concrete integration test module calls once.
//
// pub async fn run_client_suite<C: ReplicaClient>(client: C);
// (body lives in src/client/mod.rs, possibly behind #[cfg(test)])
```

**Key design note:** the Go `ReplicaClient` uses `*slog.Logger` plumbing (`SetLogger`) and passes `context.Context` as the first argument. In Rust: (a) drop `SetLogger` — backends use `tracing` spans directly; (b) `ctx` maps to a Tokio `CancellationToken` or simply cooperative Tokio task cancellation — the simplest approach is to pass no separate context and rely on `tokio::select!` + `CancellationToken` at the call sites that need cancellation, matching the project's Tokio-first design.

## 5. Footguns & Correctness-Critical Spots

**5.1 TXID sort order invariant (replica_client.go:29–32, test lines 52–113)**
The `LTXFiles` iterator MUST return files sorted by `MinTXID` ascending for a given level. The Go test writes files out-of-order and then asserts the sorted order. A backend that relies on filesystem or object-store listing order without sorting will fail this test. Sort must be stable and handle the `seek` parameter correctly: if `seek` is non-zero, the iterator starts from the first file whose `MinTXID >= seek`.

**5.2 `ErrNotExist` / `NotFound` mapping (test lines 191–198)**
The `OpenLTXFile` method MUST return an error that satisfies `errors.Is(err, os.ErrNotExist)` for a missing file. In Rust this is `io::Error` with kind `ErrorKind::NotFound`, wrapped in `crate::Error::Io`. S3 backends map `NoSuchKey` → `ErrorKind::NotFound`. Getting this wrong silently breaks the "retry vs. reset" logic in `LTXError::is_auto_recoverable` (which checks `io::ErrorKind::NotFound`).

**5.3 Timestamp preservation for PITR (test lines 247–303)**
`WriteLTXFile` must extract the LTX header's `Timestamp` field (milliseconds since epoch, `ltx-go/ltx.go:187`) and store it as the file's canonical `CreatedAt`. For object-storage backends the timestamp is stashed as object metadata (key: likely `x-amz-meta-litestream-timestamp`). When `use_metadata=true`, `LTXFiles` must fetch this metadata and populate `FileInfo.CreatedAt` from it — NOT from `LastModified`. The test asserts sub-second precision after a round-trip (±1 second tolerance, lines 296–302). The `use_metadata=false` fast path may use `LastModified` as an approximation, which is acceptable for normal sync but not PITR.

**5.4 Partial/range reads — `offset` + `size` semantics (replica_client.go:34–37)**
`OpenLTXFile` takes `(offset, size int64)`. `size == 0` means "read to end of file" (NOT zero bytes). The two-pass index fetch (`fetchPageIndexData`, lines 116–145) relies on this: it first reads `[size-32KB, EOF)` by passing `offset=size-32KB, size=0`. Implementing `size==0` as "read zero bytes" is a silent data corruption bug.

**5.5 Multipart upload for large files (test lines 307–372, 549–703)**
S3/R2 backends must use multipart upload for files > some threshold (typically 5 MB). The Go `s3.ReplicaClient` exposes `PartSize` and `Concurrency` fields. These are S3-specific (T7), not part of the core trait. However, the conformance suite should still exercise files large enough to trigger this path. The moto mock server has a known checksum bug (moto#8762) with composite multipart checksums — the test harness skips multipart tests against `127.0.0.1`. CI uses MinIO, not moto, so this distinction matters.

**5.6 `ErrStopIter` protocol (replica_client.go:16, 73)**
`FindLTXFiles` passes each `FileInfo` to the filter closure. The filter can return `(true/false, ErrStopIter)` to both include/exclude the item AND stop iteration. A non-`ErrStopIter` error stops iteration AND propagates the error. The `bool` result and the error result are both checked; returning `(true, ErrStopIter)` means "include this item and stop." Mishandling the `(match, ErrStopIter)` combination will cause incorrect retention cuts.

**5.7 `DeleteLTXFiles` atomicity**
The Go interface takes a slice of `FileInfo` and deletes them in one call. Object-store backends typically loop internally and accumulate errors. If any delete fails, the call returns an error, but some files may already be deleted. Callers (the retention GC in `store.go`) must handle partial deletion gracefully — they re-scan and retry. T5 does not need to implement retry, but must not swallow errors.

**5.8 `Init` idempotency**
`Init` must be safe to call multiple times (already-connected backends are no-ops). The Go comment at line 24–26 says "Implementations should be idempotent." Unit tests call `Init` implicitly via `RunWithReplicaClient`; tests like `TestReplicaClient_S3_BucketValidation` call `Init` directly to verify validation errors.

**5.9 Level/seek combination edge case**
When `seek > 0`, the returned iterator should start at the first file with `MinTXID >= seek`, not the first file with `MinTXID == seek`. This handles compacted ranges (e.g., seek=5 should return a file with `MinTXID=3, MaxTXID=8` because it covers TXID 5). The Go file implementation appears to rely on a directory listing from the seek point; some backends may need to list from the beginning and skip.

## 6. Suggested Sub-Step Breakdown

**Step 5a — Trait definition + constants** (~40 LOC)
Define `ReplicaClient` trait, `SNAPSHOT_LEVEL`, `DEFAULT_ESTIMATED_PAGE_INDEX_SIZE`, and `StopIter` marker. No method bodies. Compiles cleanly. Gate: `cargo build`.

**Step 5b — Utility functions: `find_ltx_files` + `fetch_ltx_header`** (~50 LOC)
Implement `find_ltx_files` (with `StopIter` handling) and `fetch_ltx_header`. These are pure compositions of trait methods and T2 LTX types; no backend needed. Gate: unit-testable with a mock client.

**Step 5c — `fetch_page_index` (two-pass tail read)** (~60 LOC)
Port `fetchPageIndexData` (replica_client.go:114–145) and `FetchPageIndex` (lines 90–98). The two-pass arithmetic is footgun territory (§5.4). Write a focused unit test with a mock client that returns a canned byte slice and verifies the page index decoded correctly. Gate: unit tests green.

**Step 5d — `fetch_page`** (~20 LOC)
Straightforward — opens a range, reads all, calls `decode_page_data`. Gate: unit test with mock client.

**Step 5e — Conformance suite skeleton** (~80 LOC)
Write `run_client_suite` (or equivalent macro) that runs the core six conformance tests (§4 critical set, excluding PITR stress tests). Parameterized over `impl ReplicaClient`. The suite compiles but the test bodies use `todo!()` stubs only until T6 provides a real client. Gate: `cargo test --no-run` passes.

**Step 5f — PITR conformance tests** (~60 LOC)
Add `PITR_ManyLTXFiles/100_Files` and `PITR_TimestampFiltering` into the suite. These require a real backend (T6 or T7); they are written in T5 but gated behind `#[cfg(feature = "integration")]` so they skip in pure unit test mode. Gate: tests compile; T6 confirms they pass.

Estimated total T5 size: ~300–350 LOC in `src/client/mod.rs` + `tests/conformance.rs`.

## 7. Size & Risk

**LOC:** The Go source is 161 LOC (interface + 5 utility functions). The Rust port will be ~300–350 LOC including async wrappers, the conformance suite skeleton, and inline doc-comments. The test file contribution to `tests/conformance.rs` adds another ~200–250 LOC when fully populated.

**Risk: MED**

Justification: The trait itself is small and mechanical; the footguns are all well-documented (sort order, ErrNotExist mapping, offset=0-means-EOF, timestamp-in-metadata). The highest-risk piece is the two-pass `fetchPageIndexData` tail-read arithmetic — an off-by-one in the `size - TrailerSize - 8 - page_index_size` expression (`replica_client.go:135`) would silently corrupt page-index lookups during restore. This is correctness-critical but isolated to ~15 lines. The conformance suite shape is also non-trivial to get right in Rust (async generics over the trait), but it is structural rather than algorithmic risk.
