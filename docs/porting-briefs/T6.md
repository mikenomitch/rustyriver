# T6 Porting Brief — `client/file.rs` (File-Backed ReplicaClient)

> Upstream spec: `reference/litestream-go/file/replica_client.go` (374 LOC)
> and `reference/litestream-go/file/replica_client_test.go` (821 LOC).
> Wave 3 task; depends on T5 (`ReplicaClient` trait + conformance suite).

---

## 1. Purpose

`FileReplicaClient` is the simplest concrete `ReplicaClient` implementation: it
reads and writes LTX files directly to a local (or NFS/FUSE) filesystem path.
There is no network I/O, no multipart logic, and no pagination; the level
directories are plain OS directories enumerated with `readdir`.  The file client
is the golden-path backend for local integration testing and for the G2
round-trip gate — every integration test that does not specifically target
MinIO runs through it.  It also implements `ReplicaClientV3`, the legacy
read-only shim for restoring v0.3.x backups (shadow-WAL/generations format);
per PLAN.md §2 ("DROP: all v0.3 machinery") this V3 surface is **out of scope**
for rustyriver and should be omitted from the port.

**Type mappings:**

| Go type | Rust type | Notes |
|---|---|---|
| `ltx.TXID` (`uint64`) | `crate::TXID` (newtype `u64`) | defined in `src/lib.rs` |
| `*ltx.FileInfo` | `LtxFileInfo` struct (T5 defines, or locally) | fields: `level: u32`, `min_txid`, `max_txid`, `size: u64`, `created_at: std::time::SystemTime` |

| `io.ReadCloser` | `Box<dyn AsyncRead + Send + Unpin>` | caller must close / drop |
| `*litestream.LTXError` | `crate::LTXError` / `crate::Error::Ltx(…)` | already in `src/error.rs` |
| `*slog.Logger` | `tracing::Span` / subscriber | no direct port; use `tracing::debug!` spans |

**Trait vs struct:** `FileReplicaClient` is a concrete struct; `ReplicaClient`
is a trait (defined in T5 as `client/mod.rs`).  There is no Tokio-specific
constraint on the struct itself; `async_trait` is used for the dyn-compatible
async methods.

**V3 surface (GenerationsV3, SnapshotsV3, WALSegmentsV3, OpenSnapshotV3,
OpenWALSegmentV3):** The Go struct implements `ReplicaClientV3`
(`file/replica_client.go:29`).  Per PLAN.md §2, all v0.3.x code is **dropped**
from rustyriver.  Do not port these five methods.  If the T5 trait does not
include V3 methods (which it should not), there is nothing to skip here.

## 3. Format-Layer Integration Points

The file client is thin — it is a filesystem shim over format types supplied by
earlier tasks.  The porter must ensure these items exist and are importable
before `client/file.rs` can compile:

| Item | Source task | Used in file client at |
|---|---|---|
| `crate::TXID` (newtype `u64`, `Display` = 16-hex) | T4 (`src/lib.rs`) | path construction, seek comparison |
| `crate::ltx_level_dir(root, level)` | T4 (`src/lib.rs:269`) | `ltx_level_dir()` helper |
| `crate::ltx_file_path(root, level, min, max)` | T4 (`src/lib.rs:280`) | `ltx_file_path()` helper |
| `crate::Error` / `crate::LTXError` / `crate::new_ltx_error` | T4 (`src/error.rs`) | `open_ltx_file` wraps OS errors in `LTXError { op: "open", … }` (file/replica_client.go:139) |
| `LtxFileInfo` struct (level, min_txid, max_txid, size, created_at) | T5 (`client/mod.rs`) | return type of `write_ltx_file`, `ltx_files` |
| `ReplicaClient` trait | T5 (`client/mod.rs`) | the impl block itself |
| LTX header parse (`peek_header` / `ltx::Header::from_bytes`) | T2 (`src/ltx.rs`) | `write_ltx_file` peeks the first `HeaderSize` bytes to extract `Timestamp` (file/replica_client.go:168-171) |

The header peek in `WriteLTXFile` is the only forward-format dependency.  The
porter needs exactly: **a function that reads the first 100 bytes from an
`AsyncRead` and returns the `Timestamp` field** (milliseconds since Unix
epoch).  If T2 is not yet landed when T6 starts, stub this as a compile-time
`unimplemented!()` protected by a `#[cfg(test)]` guard and resolve at T2 merge.
Log the dependency in `OPEN_QUESTIONS.md`.

**On-disk layout (confirmed from golden replica):**

```
<root>/
  ltx/
    <level>/           ← e.g. "0" for L0
      <min>-<max>.ltx  ← e.g. "0000000000000001-0000000000000001.ltx"
```

- `<root>` is the `path` field of `FileReplicaClient`.
- `<level>` is the decimal integer (not hex).
- Filename regex: `^([0-9a-f]{16})-([0-9a-f]{16})\.ltx$` — `ltx-go/ltx.go:484`.
- `SnapshotLevel = 9` (`compaction_level.go:9`) stores full snapshots at
  `ltx/9/<min>-<max>.ltx` — same layout, different level dir.
- No subdirectory nesting below the level dir; flat list of `.ltx` files.

The golden replica at `tests/fixtures/golden/replica/ltx/0/` confirms:
`0000000000000001-0000000000000001.ltx` through `…000006-…000006.ltx`.

## 5. Footguns & Correctness-Critical Spots

### 5.1 Atomic write: temp-file → rename → Chtimes (file/replica_client.go:182–233)

The write sequence is:

1. Write to `<filename>.tmp`.
2. `f.Sync()` — must flush to OS buffer before rename.
3. `f.Stat()` to capture actual bytes written.
4. `f.Close()`.
5. `os.Rename(tmp, final)` — atomic on POSIX; on Windows it is **not** atomic
   (requires `MoveFileExW` with `MOVEFILE_REPLACE_EXISTING`).  Tokio's
   `tokio::fs::rename` uses the OS primitive; add a comment noting the Windows
   caveat.
6. `os.Chtimes(final, timestamp, timestamp)` — sets both atime and mtime to
   the timestamp extracted from the LTX header.  This is the `CreatedAt` field
   returned in `LtxFileInfo`.

**Temp-file cleanup:** the `defer` in Go (line 192–197) removes the `.tmp`
file on **any error path**.  In Rust, use a guard struct or a `scopeguard`-
style closure.  Tests `WriteLTXFile_ErrorCleanup/*` are the red/green gates for
this.

**Critical:** if the `TeeReader` + header-peek step fails (e.g., reader is
shorter than `HeaderSize = 100` bytes), the Go code returns an error before
touching the filesystem (line 168–171).  The Rust port must do the same: peek
the header **before** creating the temp file.  Otherwise a bad writer leaves
behind an empty `.tmp`.

### 5.2 Header peek: `TeeReader` pattern (file/replica_client.go:164–175)

Go tees the reader into a `bytes.Buffer`, peeks `ltx.PeekHeader`, then
reconstructs a `MultiReader(buf, rd)` to feed the full data to the file.
In async Rust the equivalent is:

```rust
let mut header_buf = [0u8; ltx::HEADER_SIZE];
let n = reader.read_exact(&mut header_buf).await?;
let ts = ltx::peek_timestamp(&header_buf)?;
// reconstruct: chain header_buf slice with the rest of `reader`
let full = tokio::io::AsyncReadExt::chain(
    std::io::Cursor::new(header_buf.to_vec()), reader
);
```

**Do not** call `read_exact` and then discard the buffer — the header bytes
must be re-fed to the `io::copy` that writes to disk.

### 5.3 TXID ordering in LTXFiles (file/replica_client.go:113–130)

The Go code builds a `[]*ltx.FileInfo` slice and passes it to
`ltx.NewFileInfoSliceIterator`, which **sorts by (Level, MinTXID, MaxTXID)**
(`ltx-go/ltx.go:534-542`).  Files with `minTXID < seek` are filtered out
**before** inserting into the slice (line 116–118).  The sort is stable on the
slice content, not on filesystem order.

Rust must replicate this: filter first, then sort by `(level, min_txid, max_txid)`.  Do not rely on directory enumeration order (OS-dependent; HFS+/APFS
may or may not give sorted output).

### 5.4 ModTime used as CreatedAt (file/replica_client.go:127)

The file backend stores `CreatedAt` as the OS `ModTime`, which `WriteLTXFile`
explicitly sets via `Chtimes` to the timestamp from the LTX header
(line 229–231).  This means `LtxFiles` does **not** need to re-parse the LTX
header to get timestamps — it reads `ModTime` from `DirEntry`.  The
`use_metadata` parameter is a no-op for the file backend (comment on line 95).
Do not add a conditional `HeadObject`-style fetch; just use `DirEntry::modified()`.

### 5.5 `LTXError` wrapping on open failures (file/replica_client.go:139)

When `os.Open` fails, the file client does **not** return the raw OS error.
It wraps it with `litestream.NewLTXError("open", path, level, minTXID,
maxTXID, err)`.  `IsAutoRecoverable()` returns `true` for `NotFound`, `false`
for `PermissionDenied`.  Tests `OpenLTXFile_OpenErrorReturnsLTXError/*` assert
this contract directly — if rustyriver returns a raw `io::Error` here the tests
fail.  Use `crate::new_ltx_error` already defined in `src/error.rs`.

### 5.6 No size/offset handling for the file backend (file/replica_client.go:142–152)

`OpenLTXFile` accepts `offset` and `size`.  For the file client:

- `offset > 0`: `f.Seek(offset, io.SeekStart)` — implemented as
  `tokio::fs::File::seek(SeekFrom::Start(offset))`.
- `size > 0`: wrap the file in `io::Take(size)` — Go uses
  `internal.LimitReadCloser`.
- Both zero: return the file as-is.

Do not conflate these: `size == 0` means "read to EOF", not "read zero bytes".

### 5.7 Concurrency

The file client is **stateless** (no shared mutable state besides the
immutable `path` field).  Concurrent `WriteLTXFile` calls for the same
`(level, min, max)` tuple are safe because each call races only on the
rename: the last rename wins, and both callers write the same content (TXID
is immutable for a given transaction).  This matches the Go behavior implicitly.
No internal `Mutex` is needed.

### 5.8 `DeleteLTXFiles` silently ignores NotFound (file/replica_client.go:243–244)

```go
if err := os.Remove(filename); err != nil && !os.IsNotExist(err) {
    return err
}
```

Rust: `match tokio::fs::remove_file(path).await { Err(e) if e.kind() == io::ErrorKind::NotFound => {} ... }`.

### 5.9 `DeleteAll` uses `os.RemoveAll` not `os.Remove` (file/replica_client.go:252)

`tokio::fs::remove_dir_all` is the Rust equivalent.  Also silently ignore
`NotFound` (line 253).

## 7. Size & Risk

**Go source:** `file/replica_client.go` = 374 LOC;
`file/replica_client_test.go` = 821 LOC.

**Estimated Rust port:** ~200–250 LOC (async boilerplate is more verbose; V3
methods are dropped; conformance tests already live in T5).

**Risk: LOW**

The file client has no network calls, no retries, no multipart logic, and no
shared mutable state.  The only non-trivial parts are the `TeeReader` header-
peek pattern (§5.2) and the temp-file cleanup guard (§5.1), both of which have
direct Go test coverage.  The dependency on T2 (`ltx::peek_timestamp`) can be
stub-gated if T2 is not yet merged.  The main failure mode for this task is
an incomplete `LTXError` wrapping on `OpenLTXFile` — the conformance tests
catch it.
